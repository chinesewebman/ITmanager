# ITmanager 优化路线图 (2026-06-16)

> 主线：**便于用户操作 + 便于诊断网络故障**。
> 总原则：每条 PR 都是「编写代码 → 代码审查 → 单元测试」一条龙，绝不写一半。

---

## 0. 现状盘点（基线）

| 层 | 现状 | 优势 | 短板 |
|---|---|---|---|
| 资产 | NetBox 同步 + 自建表 + 双数据源 | 全量数据 | 字段分裂，资产详情要查两表 |
| 监控 | Zabbix 拉取 problem + history | 集成快 | 指标没本地缓存，Zabbix 挂了盲 |
| 告警 | rule → match → 通知 | 走通链路 | 抑制/升级/值班组 缺失 |
| 工单 | 自建 CRUD + GLPI 同步 | 双源 | 状态机弱，缺少 SOP 模板 |
| 可视化 | Ant Design + ECharts 雏形 | 基础够 | 没拓扑图/热力图/时间轴 |
| 后端 | Gin + GORM + Postgres | 测试 401 / 61.2% | `internal/database` 0%、`middleware` 36.5% |
| 前端 | React + Zustand | 测试 53 | 无 e2e、无 storybook |

## 1. 关键问题（影响"网络故障诊断"主流程）

- 🔴 **缺：全栈诊断时间线** —— `GET /api/v1/diagnostics/{asset_id}/timeline` 端点不存在
- 🔴 **缺：网络拓扑** —— `asset_networks.parent_id` 有数据但前端没渲染
- 🟡 **弱：抑制/降噪** —— Zabbix 同一 host 1 分钟 20 条 problem 全部进库
- 🟡 **弱：值班 + 升级** —— 通知渠道是平的 webhook/email/钉钉群

## 2. 优化方案（按 ROI 排序）

### 🥇 P0-1：诊断时间线聚合端点（**2-3h**）
- **后端**：`internal/diagnostics` 包，1 个 SQL（pg `UNION` 6 表）+ 1 个 handler
- **前端**：`/diagnostics/:assetId` 路由，Ant Design `Timeline` 组件
- **测试**：handler 8-10 + service 4-5（mock DB 6 表 JOIN）
- **价值**：1 端点解决"故障在哪 + 历史发生过啥"两个问题
- **状态**：⏳ 待启动

### 🥈 P0-2：抑制规则引擎（**4-6h**）
- **后端**：`internal/service/alert_suppression.go` + 表 `alert_suppressions(severity_max, host_pattern, time_window, ttl)`
- 落库前 in-memory 抑制器 + Redis pub/sub
- **前端**：设置页加"抑制规则 CRUD"
- **测试**：service 12-15（5 分钟同 host 合并 / 阈值匹配 / TTL 过期 / 降级）
- **价值**：告警量 -70%
- **状态**：⏳ 待启动

### 🥉 P1-1：网络拓扑可视化（**6-8h**）
- 用 `react-flow`（比 d3/three.js 简单 10x）
- **后端**补 `GET /api/v1/topology` 返回 `nodes[]/edges[]`（从 `asset_networks` 聚合）
- **前端**：拓扑页 + 节点点击进资产详情 + 故障节点高亮（红色 + 告警数 badge）
- **测试**：handler 6-8 + 前端 4-5（react-flow 渲染 + 节点点击）
- **价值**：故障定位时间从"翻列表" → "看图"
- **状态**：⏳ 待启动

### P1-2：值班 + 升级（**8-10h**）
- `oncall_schedules` 表 + `oncall_shifts` + `escalation_policies`
- 简单轮值 = 1 个 cron 每天切一次 owner
- 升级 = ack 超时 → 通知 next level
- **测试**：service 15-20（轮值边界 / 时区 / 升级路径 / 兜底 owner）
- **状态**：⏳ 待启动

### P2-1：故障剧本 Runbook（**3-4h**）
- 资产 → 关联 SOP（重启/切换/排错步骤）
- 告警详情页 → 侧栏显示 runbook
- 表 `runbooks(asset_type, content_md, steps_json)`
- **测试**：service 8-10（CRUD + 资产类型匹配 + 步骤渲染）
- **状态**：⏳ 待启动

### P2-2：Zabbix 兜底（**4h**）
- 1 张表 `metric_snapshots(asset_id, key, value, ts)` 落本地 TimescaleDB
- 定期 cron 从 Zabbix 拉最新值
- Zabbix 挂了用本地快照
- **测试**：cron 6-8 + 查询 service 4-5
- **状态**：⏳ 待启动

## 7. 流程（每条 PR）

每条优化项**必须**走完三步才标 ✅：
1. **编写代码** — 实现功能 + 1 次自查
2. **代码审查** — 至少看 1 遍：边界条件 / 错误处理 / 并发 / 资源泄漏
3. **单元测试** — 覆盖正常路径 + 边界 + 失败路径 + 并发（-race）

每个 PR 包含：
- 功能代码
- 测试代码（覆盖率不能下降）
- commit message 写明「优化项 ID + 三大点」
- pre-commit hook 自动跑（gofmt + swagger validate + .bak 拦截）
